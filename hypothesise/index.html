Rubber Duck Debugging

Step 2: Hypothesise
Once you have found your problematic code chunk or problematic input, your goal is to think about all the ways in which this code chunk could fail in the way that you're seeing!
When you were reproducing the error, you used 'print' statements to identify the earliest line where what you're expecting the code will do is not what the code has done. Now your job is to figure out why the code is interpreting your instructions in this unexpected way. This means examining any functions you used in that line, and thinking about all the ways in which it could throw this kind of output instead of the one you're expecting.
Don't be afraid to throw out lots of wild ideas, this is just the ideas generation stage. The more potential problems you've thought about, the more likely that your actual problem is part of this set.
